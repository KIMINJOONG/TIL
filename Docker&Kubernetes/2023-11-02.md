# 수동배포에서 관리형 서비스로

- Do-it-Yoruself접근 방식에는 몇가지의 잠재적인 단점이 존재한다.

  - 때로는 완전한 통제와 그에 따른 책임을 원하는지, 아니면 통제가 덜한 상태로도 괜찮은지 선택이 필요
    - 이는 완전히 자체 관리되는 서비스 혹은 자체 관리되는 리모트 호스트에서 써드파티 솔루션으로 전환하는것.
    - 예를들어, 자체 EC2인스턴스를 실행하는것처럼 자체 머신을 실행하는 대신 관리형 서비스를 사용할수있다.
    - 왜냐면 우리 자체 머신, 자체 리모트 호스트를 지속적으로 모니터링하고 최신상태로 유지해야하고, 확장해야한다. 애플리케이션이 커지고 들어오는 트래픽이 커지면 우리 스스로 어떻게든 처리하고 애플리케이션이 다운되지않도록 보장해야함(지식이 있으며, 경험이 풍부한 관리자, 클라우드 전문가 그렇지않은 경우에는 위험한 설정을 할수도있음)

- 그렇기때문에 관리형 리모트 머신이라 부르는방식으로 진행

# ECS

- Amazon에서 제공하는 서비스로 Elastic Container Service의 약자로 이름에서 알수있듯 컨테이너 관리를 도와주는 서비스, 컨테이너 구동 및 실행, 모니터링 등등

## 장점

- 전체 생성관리, 업데이트, 모니터링, 스케일링 이 모든것이 단순화 된다.
- AWS와 같은 써드 파티 서비스는 이러한것들을 처리하며 우리가 해야할것은 AWS가 관심을 가져야할것들을 알리는것뿐.
- 따라서 단순히 애플리케이션이나 컨테이너를 배포하려는 경우라면 이러한 서비스를 사용하는것이 좋다.
- 모든 세부 설정 작업에대해 걱정할 필요 x
- 더 작아진 통제로 책임이 작아지지만 엉망이 될 우려도 작아짐
- 이는 쉽게 말하자면, 컨테이너 배포 및 컨테이너 실행이 더 이상 도커 명령으로 수행되지 않음을 의미
- 특정 시스템에 도커를 더 이상 설치 x 이것이 관리형 서비스의 배경사상
